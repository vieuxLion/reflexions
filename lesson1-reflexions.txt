comment l'utilisation de l'historique permet d'améliorer mes compétences de programmeur:
retourner rapidement à une version qui marche et comparer les différences
sécuriser les changements par retour arrière facile

quand sauvegarder les changements?
lorsque l'on choisit de le faire

commit size:
a week - trop grand, commit chaque soir
3 typos : pas un seul => les 3 ok
tt changements logiques pendant une heure => OK
deux petits bugs sur 2 fonctions différentes => plutot un par un

les avantages de choisir manuellement les commits:
- nous seuls avons la connaissance des changements effectués et de leur regroupement logique 
- inconvénient, si l'on ne committe pas pendant longtemps et que l'on perd le travail

comment utiliser GIT pour suivre les évols du logiciel?
avec GIT log, lister les evols
avec git diff, comparer les IDs et voir les lignes insérées ou détruites


pourquoi certains systèmes de gestion de version proposent des commits à plusieurs fichiers alors que d'autres proposent seulement des commit à un fichier ?
selon les cas d'usage, les modifications portents sur plusieurs ou un fichier
pour le code, généralement, plusieurs fichiers sont impactés par une modif
